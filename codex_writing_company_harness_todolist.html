Codex 기반 AI 글쓰기 회사 Harness To-do List

이 프로젝트는 웹앱이 아니다. AI API를 직접 호출하지도 않는다. ChatGPT 계정으로 로그인된 Codex를 “글쓰기 직원”으로 사용하고, 로컬 하네스가 작업지시서 생성, 역할 분배, 파일 저장, 검수 흐름을 관리하는 터미널 기반 글쓰기 회사 시스템을 만든다.

핵심: Codex OAuth UI: 터미널/CLI 중심: Harness 산출물: 완성 원고 파일 미래: CrewAI 자동화

0. 프로젝트 정의

정확한 목표를 먼저 고정한다. 이 프로젝트는 “AI 글쓰기 회사 운영 프로그램”이다.

1
사용자는 CEO

장르, 방향, 승인, 수정 요청만 한다. 직접 원고를 쓰지 않는다.

2
하네스는 회사 운영 시스템

작업지시서 생성, 단계 관리, 파일 저장, 다음 작업 지시를 담당한다.

3
Codex는 실행 직원

ChatGPT OAuth 로그인 상태의 Codex가 로컬 파일을 읽고 원고 파일을 작성한다.

최종 목표: 터미널에서 npm run company 또는 python main.py 실행 → “새 책 만들기 / 기획 / 목차 / 집필 / 편집 / 검수 / 최종본” 메뉴 선택 → 하네스가 Codex에 작업을 맡기고 결과 파일을 받아 책 완성본을 만든다.

1. 시스템 작동 흐름

AI API 호출 대신, 하네스가 Codex에게 파일 기반 작업을 시키는 구조다.

CEO 명령 입력터미널 질문
하네스 상태 확인project.json
작업지시서 생성tasks/current.md
Codex 실행codex exec
원고 파일 작성projects/book
검수 로그 저장logs
CEO 승인 요청수정/진행
최종본 패키징final.md
중요: 하네스가 GPT 모델을 직접 부르는 것이 아니다. 하네스는 Codex가 읽을 명령서 파일을 만들고, Codex CLI를 실행시키며, Codex가 만든 결과 파일을 정리한다.

2. 글쓰기 회사 직원 구성

각 직원은 별도의 AI API 호출 객체가 아니라, Codex에게 부여하는 역할 프롬프트와 작업지시서다.

직원 역할 하네스에서의 구현 결과 파일
CEO 사용자. 방향, 승인, 취향 판단 터미널 입력 / 승인 메뉴 project.json의 decisions
Chief Producer 전체 작업 순서 결정 prompts/producer.md + workflow next_action.md
기획자 작품 컨셉, 독자층, 세계관 설계 prompts/planner.md story_bible.md
구성 작가 목차, 챕터별 사건, 갈등 구조 설계 prompts/outliner.md outline.md
메인 작가 실제 본문 초안 작성 prompts/writer.md chapters/ch001_draft.md
편집자 문장, 몰입도, 장면 연결 수정 prompts/editor.md chapters/ch001_edited.md
설정 검수자 인물, 시간선, 세계관 오류 점검 prompts/continuity_checker.md reviews/ch001_continuity.md
최종 편집자 전체 원고 통합, 최종본 정리 prompts/finalizer.md final/book_final.md

3. 구축 To-do List

초보 기준으로 이 순서대로 진행하면 된다. 처음 목표는 “Codex가 파일을 읽고 쓰는 글쓰기 회사 MVP”다.

  1. 1단계: Codex OAuth 작업환경 확인

    API key가 아니라 ChatGPT OAuth 로그인 상태로 사용하는 것이 핵심이다.
    예: README.md를 만들고 “이 파일에 한 줄 추가해줘”라고 시킨다.
    나중에 하네스가 Codex를 자동 실행하려면 필요하다.
    예: results/test_result.md에 답을 저장하게 한다.

  2. 2단계: 프로젝트 폴더 구조 만들기

    이 폴더가 회사 사무실이다. 모든 작업은 이 안에서만 진행한다.
    harness, prompts, tasks, projects, results, logs, templates
    “이 프로젝트는 웹앱이 아니며, Codex OAuth 기반 로컬 글쓰기 회사다”라고 명시한다.
    현재 진행할 작업과 완료된 작업을 계속 기록한다.

  3. 3단계: 하네스의 최소 기능 정의

    project.json을 읽고 현재 단계, 최신 파일, 승인 여부를 관리한다.
    CEO 입력 + 역할 프롬프트 + 기존 원고를 합쳐 tasks/current_task.md를 만든다.
    하네스가 터미널 명령으로 Codex를 실행한다.
    Codex가 만든 파일을 확인하고 최신 결과를 화면에 요약해 보여준다.
    언제 어떤 역할로 어떤 작업을 실행했는지 logs에 남긴다.

  4. 4단계: 직원별 프롬프트 카드 만들기

    전체 작업 순서와 다음 작업 추천을 담당한다.
    작품 컨셉, 장르, 주인공, 독자층, 세계관을 설계한다.
    전체 목차와 챕터별 사건 구조를 만든다.
    초안을 작성하되, 반드시 story_bible과 outline을 따른다.
    초안을 더 읽기 좋게 수정한다.
    설정 오류와 모순을 찾아낸다.
    최종 원고를 통합하고 형식을 정리한다.

  5. 5단계: 터미널 메뉴형 하네스 만들기

    초보라면 Python 또는 Node.js 중 익숙한 하나만 선택한다.
    1. 새 책 만들기 2. 기획하기 3. 목차 만들기 4. 챕터 쓰기 5. 편집하기 6. 검수하기 7. 최종본 만들기
    장르, 분위기, 목표 분량, 원하는 소재, 금지사항을 터미널에서 입력받는다.
    사용자가 “기획하기”를 고르면 planner 역할의 작업지시서가 생성된다.
    예: “기획서가 생성되었습니다. 다음은 목차 만들기를 추천합니다.”

  6. 6단계: Codex 실행 연결

    예: run_codex(task_file)
    “tasks/current_task.md를 읽고, 지시된 출력 파일에 결과를 작성하라.”
    Codex 실행 후 지정된 파일이 만들어졌는지 확인한다.
    파일이 없거나 형식이 틀리면 “다시 작성하라”는 correction task를 만든다.

  7. 7단계: 첫 번째 글쓰기 파이프라인 완성

    projects/book_001/project.json 생성.
    장르, 주인공, 세계관, 말투, 금지사항이 정리된다.
    5화 또는 10화 단위 목차부터 시작한다.
    chapters/ch001_draft.md
    chapters/ch001_edited.md
    reviews/ch001_continuity.md
    chapters/ch001_final.md

  8. 8단계: 품질관리 규칙 만들기

    Codex가 엉뚱한 곳에 글을 쓰지 않도록 한다.
    draft, edited, checked, final을 구분한다.
    새 설정이 생기면 반드시 story_bible.md에 반영한다.
    사용자가 금지한 소재, 톤, 표현을 검수한다.
    설정 오류, 문체 일관성, 챕터 연결, 미완성 문장 확인.

  9. 9단계: 반복 생산 구조 만들기

    ch002_draft → ch002_edited → ch002_final 순서.
    다음 화 작성 시 전체 원고를 다 읽게 하지 말고 요약을 제공한다.
    각 화가 draft인지 final인지 추적한다.
    “주인공 성격을 더 냉정하게”, “분위기를 어둡게” 같은 변경을 반영한다.

  10. 10단계: 최종본 패키징

    각 챕터 final 파일을 합쳐 final/book_final.md 생성.
    출판사 느낌이 나도록 패키지화한다.
    처음에는 md/txt만 충분하다. docx/pdf는 나중에 붙인다.
    작업 중간 결과가 사라지지 않게 한다.

  11. 11단계: 미래 CrewAI 자동화 대비

    나중에 CrewAI Agent로 바꾸기 쉽게 한다.
    planning → outlining → drafting → editing → checking → finalizing.
    초기에는 수동 승인, 나중에는 자동 승인 모드.
    지금 구현하지 말고, 나중에 어떤 구조로 옮길지만 기록한다.

5. Codex에게 바로 넣을 첫 명령

아래는 Codex에게 프로젝트 뼈대를 만들게 하는 첫 프롬프트다.

나는 바이브코딩 경력만 있는 초보자다.
웹앱이 아니라, Codex 안에서 하네스를 활용해 돌아가는 'AI 글쓰기 회사'를 만들고 싶다.

중요한 조건:
- OpenAI API를 직접 사용하지 않는다.
- ChatGPT 계정으로 OAuth 로그인된 Codex를 글쓰기 직원처럼 사용한다.
- 하네스는 Codex가 읽을 작업지시서 파일을 만들고, Codex 실행을 도와주고, 결과 파일을 정리한다.
- 사용자는 CEO 역할이다.
- 사용자는 직접 원고를 쓰지 않고, 장르/방향/승인/수정 요청만 한다.
- 처음에는 완전 자동화하지 않는다.
- 나중에는 CrewAI 같은 멀티에이전트 자동화로 확장할 수 있게 구조를 잡는다.

1차 목표:
- Python 기반 터미널 메뉴형 MVP를 만든다.
- 메뉴는 다음과 같다.
  1. 새 책 만들기
  2. 기획서 생성
  3. 목차 생성
  4. 챕터 초안 생성
  5. 편집본 생성
  6. 설정 검수
  7. 최종본 생성
  8. 현재 상태 보기
- 각 메뉴는 실제 글을 직접 생성하지 말고, 먼저 tasks/current_task.md 작업지시서를 생성하게 한다.
- Codex 실행 부분은 harness/codex_runner.py에 함수로 분리한다.
- 지금 당장은 안전하게 dummy 모드도 지원하게 한다.
- 나중에 codex exec로 연결할 수 있도록 TODO 주석을 남긴다.

- 초보자가 이해할 수 있도록 주석을 많이 달아라.

- 각 메뉴가 어떤 파일을 만들고 읽는지 설명하라.
- OpenAI API 키를 요구하지 마라.
- Codex OAuth 로그인으로 작동하는 구조라는 점을 명확히 적어라.

6. 1차 MVP 성공 기준

하네스 실행

터미널에서 메뉴가 뜨고, 사용자가 작업을 선택할 수 있다.

작업지시서 생성

선택한 역할에 맞는 current_task.md가 자동 생성된다.

Codex 연결 준비

codex_runner.py에서 Codex 실행부를 분리해 둔다.

파일 기반 회사

기획서, 목차, 원고, 검수본이 폴더에 남는다.

진짜 1차 목표: “완성된 소설”이 아니라, Codex가 읽을 작업지시서를 하네스가 안정적으로 만들고, 결과 파일을 회사 폴더 구조에 쌓는 것이다.

7. 진행 순서 요약

  1. Codex OAuth 로그인이 정상인지 확인한다.
  2. Codex가 로컬 파일을 읽고 쓰는지 테스트한다.
  3. Python 터미널 메뉴형 하네스를 만든다.
  4. 역할별 프롬프트 파일을 만든다.
  5. 메뉴 선택 → 작업지시서 생성까지 완성한다.
  6. 작업지시서를 Codex에게 실행시키는 구조를 붙인다.
  7. 기획서 → 목차 → 1화 → 편집 → 검수 → 최종본 순서로 확장한다.
  8. 5화 정도 안정화되면 자동 반복 기능을 붙인다.
  9. 그다음 CrewAI 등 멀티에이전트 자동화를 검토한다.
